A1 · CRIT-3 OTP leak in response body — claimed fixed, NOT FIXED
Task #280 "Done looks like": "OTP flow never returns the code in the HTTP response body when NODE_ENV=production, regardless of SMTP config"
Current code at `artifacts/api-server/src/routes/auth.ts:222-225`:
```ts
  // SMTP not configured at all — return code in response for dev/internal use
  return res.json({ ok: true, code, message: "SMTP not configured. Code shown for development use only." });
});
```
Problem: No `NODE_ENV` check. The OTP code is still returned in the response body in every environment, including production, whenever SMTP env vars are missing. This is the original critical vulnerability unchanged.
Required change: Wrap the fallback branch so the code is returned only when `NODE_ENV !== "production"`. In production with no SMTP, return HTTP 503 with an operator-facing message and do not log the code server-side either (logs ship and persist).
```ts
  // SMTP not configured at all
  if (process.env.NODE_ENV === "production") {
    req.log?.error(
      "OTP requested in production with SMTP entirely unconfigured — refusing to expose code. " +
      "Set SMTP_HOST, SMTP_USER, SMTP_PASS in Replit Secrets."
    );
    return res.status(503).json({
      error: "Login is temporarily unavailable. Email delivery is not configured. Contact your administrator."
    });
  }
  // Dev/staging convenience
  return res.json({ ok: true, code, message: "SMTP not configured. Code shown for development use only." });
```
Do NOT log the OTP code at `info` or `warn` level anywhere — only at `debug` and only in non-production.

A2 · HIGH-6 AI endpoints return HTTP 200 on provider failure — claimed fixed, NOT FIXED
Task #281 "Done looks like": "AI endpoints return HTTP 502 for provider unreachable / 401 key errors, not HTTP 200 with answer: null"
Current code at `artifacts/api-server/src/routes/ai.ts` — every error path still uses `res.json(...)` with implicit HTTP 200. Sampled lines (full count: 19 sites):
```
518:  return res.json({ answer: null, setupRequired: true, error: "OPENAI_API_KEY is not configured" });
735:  return res.json({ suggestions: null, error: "Failed to parse AI response" });
741:  return res.json({ suggestions: null, error: "Failed to parse AI response" });
776:  return res.json({ brief: null, setupRequired: true, billingRequired: false });
816:  return res.json({ brief: null, setupRequired: true, billingRequired: false });
990:  return res.json({ result: null, error: "No valuation entries found for this target" });
1074: return res.json({ result: null, error: "Failed to parse AI response" });
1139: return res.json({ result: null, error: "Insufficient diligence items (need ≥ 3 items across workstreams)" });
1216: return res.json({ result: null, error: "Failed to parse AI response" });
```
Problem: Monitoring cannot distinguish a real provider outage (dead API key, lapsed billing, OpenAI 5xx) from a legitimate "not configured" or "insufficient input" response. All look identical from outside.
Required change: Distinguish three cases and use correct HTTP status:
Case	HTTP	Example
Provider/system failure (OpenAI 401, 429, 5xx; parse failure; network error)	502	`res.status(502).json({ error: "AI provider error", providerError: <safe summary> })`
Rate-limit passthrough from OpenAI	429	`res.status(429).json({ error: "AI provider rate-limited" })`
Application-level "not configured" or "insufficient input"	200 (advisory)	Keep as-is — these are legitimate pre-flight states
Success	200	Keep as-is
Scope: All 19 `return res.json(...)` lines in `routes/ai.ts` need to be triaged into one of the three buckets. Don't blanket-change them all to 502 — that would break the legitimate "not configured" advisory path that the frontend relies on.

Required: Add a dedicated AI rate limiter — suggest 10 req/min per user for write endpoints (`/ask`, `/meeting-notes`, `/opportunity-brief`, `/weekly-brief`, `/valuation-sanity`, `/dd-synthesis`); read endpoints can stay on the general limiter.
