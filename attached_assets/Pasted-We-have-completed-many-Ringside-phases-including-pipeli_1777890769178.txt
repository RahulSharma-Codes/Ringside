Part 1 — Deal Journey / Progressive Data UX

Problem:
At teaser/origination stage, users do not know synergy score, financial attractiveness, risk penalty, diligence readiness, and other later-stage fields. Ringside should not force users to fill or interpret those fields too early.

Goal:
Make the opportunity journey stage-aware.

Required behavior:
1. Add a stage-aware target detail experience.
2. Early-stage opportunities should show core teaser/origination fields prominently:
   - target code
   - project name
   - legal name if known
   - sector
   - geography/country
   - source / sourcing channel
   - owner
   - priority tier
   - initial strategic rationale
   - next action
3. Diligence/scoring fields should be visually marked as:
   - Not assessed yet
   - Pending diligence
   - Not applicable at current stage
4. Do not show missing scores as 0 unless the database truly stores a real score of 0.
5. If score fields are null/empty, display “Not assessed” rather than a numeric score.
6. Add short helper text explaining that deeper scoring becomes meaningful once diligence begins.
7. Make the New Opportunity form stage-aware:
   - default stage should be Origination or Initial Screening
   - do not require synergy/risk/financial score fields at creation
   - let users add these later
8. Add progressive sections:
   - Teaser / Origination Snapshot
   - Screening View
   - Diligence Assessment
   - Offer / Integration Readiness
9. On the dashboard/pipeline, score summaries should handle missing scores gracefully:
   - show “Assessment pending”
   - do not drag averages down because scores are missing
   - average only assessed scores where possible

Goal:
Move from single APP_PASSWORD login toward individual email/password users and roles.

First inspect the current auth implementation.

Preferred long-term approach:
Use Supabase Auth email/password with a roles/profile table.

Do not remove APP_PASSWORD login immediately.
Implement in a safe staged way.

Required:
1. Propose the authentication architecture first before coding.
2. Determine whether Supabase Auth can be integrated cleanly with the current Express API + React app.
3. Proposed roles:
   - Admin
   - Deal Lead

   - Executive Viewer
4. Define permissions:
