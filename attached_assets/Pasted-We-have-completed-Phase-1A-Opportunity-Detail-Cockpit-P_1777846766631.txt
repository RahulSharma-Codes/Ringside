Phase 4B goal:
Add a diligence workspace that helps users manage due diligence readiness for each opportunity and across the pipeline.

Required features:

1. Target Detail Diligence Tab
Add a Diligence tab or section to the target detail page.

Show diligence workstreams:
- Commercial
- Financial
- Legal
- Tax
- HR
- Technology
- Operations
- Integration

Each workstream should show checklist/action items related to diligence:
- description/title
- owner
- due date
- priority
- status
- notes/description
- linked opportunity
- workstream/category

If the existing actions schema can support this with existing fields, reuse actions.
If it cannot support workstream/category cleanly, stop and propose the smallest schema addition instead of applying migrations automatically.

2. Add Diligence Item
From the target detail Diligence tab, user can add a diligence item:
- workstream
- title/description
- owner
- due date
- priority
- status
- notes

3. Edit Diligence Item
User can edit diligence item fields.
Use existing action update logic if diligence items are represented as actions.

4. Complete/Reopen Diligence Item
User can mark a diligence item complete or reopen it.
Reuse existing action status update logic where possible.

5. Diligence Readiness Score
For each target, calculate a readiness score:
- percentage of diligence items completed
- show blocked count
- show overdue count
- show missing workstreams with no items

Display this on the Diligence tab and, if easy, on target overview.

6. Global Diligence Review Page
Add a new page called Diligence Review.

Show:
- targets with blocked diligence items
- targets with overdue diligence items
- Must-Win targets with incomplete diligence
- diligence completion percentage by target
- missing workstreams by target
- recently completed diligence items

Each card should link to the target detail page.

7. Navigation
Add "Diligence Review" to the sidebar/nav.

8. Mobile-first UI
- Card-based layout
- No cramped tables
- Readable on 375px screen
- Use workstream chips, status badges, priority badges, overdue badges

9. Backend/API
Inspect existing actions and targets routes first.
Extend existing routes where appropriate.
Add read/write endpoints only if needed.
All data must come from Supabase.
No mock data.
No demo data.
